## AJAX

Ajax steht für Asynchronous JavaScript and XML. Es stellt das grundprinzip für Websiten dar, Daten im Hintergrund nachzuladen. Heutzutage wird dafür aber meistens kein XML, sondern JSON verwendet. Üblicherweise fragt man Daten von einer REST-API ab, damals war auch SOAP beliebt, wird heute aber kaum noch verwendet.

Der "klassische Weg" Ajax zu verwenden, ist per `XMLHTTPRequest`:

```js
const req = new XMLHttpRequest();

req.open("GET", "www.example.org?param=value", true);

req.onreadystatechange = function receive() {
    if (req.readyState == 4) answer = req.responseText;
}

req.send();
```

Vor einer ganzen Weile ist dann die `fetch` API dazu gekommen und steht in jedem Browser zur Verfügung:

```js
await (await fetch("example.com")).json();
```

(Sowohl fetch wie auch `#json()` geben einen Promise zurück)

## Websockets

Wenn man eine Website läd, fragt man vom Server eine Resource an, und bekommt diese zurück. Der Server kann also nicht einfach Daten schicken, die der Client gar nicht angefragt hat (Mit Ausnahmen, z.B. HTTP 3). Mit Websockets kann man eine Verbindung zwischen Client und Server aufbauen auf der dann beide Seiten Nachrichten senden können. Dies ermöglicht z.B. das senden von Events oder anderen Echtzeitanwendungen.

Der Overhead von Websockets ist dabei auch geringer, da nur eine HTTP Anfrage gesendet werden muss und dann auf das Websocket Protokoll umgestellt wird.

In Java kann man einfach einen `ServerEndpoint` erstellen, um Websocket Connections anzunehmen:

```java
@ServerEndpoint("/helloSocket")
public class HelloSocket {
    @OnMessage
    public void onMessage(Session session, String message) throws IOException {
        session.getBasicRemote().sendText("hello Client");
    }

    @OnOpen
    public void onOpen(Session session) throws IOException { /* ... */ }

    @OnClose
    public void onClose(Session session) throws IOException { /* ... */ }

    @OnError
    public void onError(Session session, Throwable throwable) { /* ... */                        }
}
```

Auf JavaScript Seite ist es genau so einfach:

```js
const socket = new WebSocket(myURL)
socket.onopen = function () { /* ... */ }
socket.onmessage = function (messageEvent) { /* ... */ }
socket.onerror = function (errorEvent) { /* ... */ }
socket.onclose = function (closeEvent) { /* ... */ }
socket.send("Hello Server")
```

## REST Services

Die implementierung einer REST-API in Java ist vergleichbar einfach. Folgende Dependencies muss man dennoch dem Projekt hinzufügen: `jakarta.ws.rs:jakarta.ws.rs-api:4.0.0`, `org.glassfish.jersey.containers:jersey-container-servlet:3.1.11`, `org.glassfish.jersey.inject:jersey-hk2:3.1.11`, `org.glassfish.jersey.media:jersey-media-json-jackson:3.1.11`.

Die Grundlage für eine REST Anwendung ist die `Application`:

```java
import jakarta.ws.rs.ApplicationPath;
import jakarta.ws.rs.core.Application;

@ApplicationPath("rest")
public class MyApplication extends Application {}
```

Sie definiert den Basispfad der Anwendung, in diesem Fall `/rest`.

Einzelne Ressourcen (Endpoints) definiert man dann mit den Annotations von `jakarta.ws.rs`:

```java
@Path("hello")
@Produces(MediaType.TEXT_HTML)
public class HelloWorldService {
    @GET
    public Response helloWorld() {
        return Response.ok("Hello World!").build();
    }
}
```

In diesem Fall die Route `GET /rest/hello`, welche "Hello World!" zurück gibt.

Mit anderen `MediaType` können andere Formate zurück gegeben werden und mit anderen HTTP Verben lassen sich auch zusätzliche Endpoints definieren:

| Verb     | Funktion   |
| -------- | ---------- |
| @ GET    | Suchen     |
| @ POST   | Erstellen  |
| @ DELETE | Löschen    |
| @ PUT    | Bearbeiten |

Es gibt noch weitere Verben, die aber bei REST typischerweise nicht verwendet werden.

Jakarta kann auch Objekte automatisch zu JSON konvertieren:

```java
@GET
public Response getAddress() {
    Address a = new Address("Mustermann", "Erika",
            "Erika.Mustermann@nowhere.de",
            "0177777777");
    return Response.ok(a).build();
}
```

Den Pfad der Routen in einer Ressource kann man auch anpassen und/oder Query Parameter oder Path Parameter abfragen:

```java
@Path("param")
@Produces(MediaType.TEXT_HTML)
public class ParamService {
    @GET
    public Response noparam() {
        return Response.ok("Kein Parameter").build();
    }

    @GET
    @Path("/{i}")
    public Response pathparam(@PathParam("i") int i) {
        return Response.ok("PathParam "+i).build();
    }

    @GET
    @Path("/param")
    public Response queryparam(@QueryParam("i") @DefaultValue("5") int i) {
        return Response.ok("QueryParam "+i).build();
    }
}
```
