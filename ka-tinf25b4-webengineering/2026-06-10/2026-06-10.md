## HTTP

HTTP steht für Hypertext Transfer Protocol und ist das Protokoll dass zur Übertragung von Daten im WWW verwendet wird.

Es basiert auf dem Anfrage und Antwort Prinzip, es muss also ein Client einen Server anfragen, um Daten zu bekommen. Der Server kann (mit Ausnahmen in HTTP 3) nicht einfach Daten an einen Client schicken, ohne dass der danach gefragt hat. 

Dem HTTP Protokoll liegt das TCP Protokoll zur Grunde, welches verschlüsselt oder unverschlüsselt Daten übertragen kann. Im Gegensatz zu UDP stellt dieses sicher, dass Daten komplett und in der richtigen Reihenfolge beim Client ankommen.

HTTP ist zustandslos. Es behandelt also alle Requests unabhängig voneinander, seidenn im Request enthalten sich Informationen. Ein Request ist in Header und Content aufgebaut:

```
<Methode> <URI> <Protokoll>
<Header>

<Content>

```

z.B.:

```
GET / HTTP/1.1
Host: developer.mozilla.org
Accept: */*
```

**Methoden**

Es gibt einige HTTP Methoden, auch HTTP Verben genannt:

- `GET`: Anforderung einer Ressource anhand der URI
- `POST`: Senden von Daten, Parameter auch über URI
- `HEAD`: Nur Header Anfragen
- `PUT`: Ressource hochladen
- `DELETE`: Ressource löschen
- `PATCH`: Ressource modifizieren
- `TRACE`: Anfrage zurücksenden (Veränderung bei Transport?)
- `OPTIONS`: Liste der Unterstützen Features und Methoden
- `CONNECT`: Vorbereitung SSL-Tunnel im Proxy

Und noch ein paar mehr. 

Eine URI ist spezifiziert nach RFC 3986 und hat den folgenden Aufbau:

`<Schema>://<Server>:<Port>/<URL-Pfad>?<Anfrage>#<Fragment>`

Im Web wird hauptsächlich das Schema `https` (verschlüsselt) verwendet, in seltenen Fällen `http` (unverschlüsselt).

**Header**

Die Header sind prinzipiell `<header>: <wert>` Freitext. Es gibt aber einige standartisierte anerkannte Header:

- host: Aufgerufener Server 
- accept: Liste der akzeptierten Dokumenttypen (MIME, z.B. text/html)
- accept-language: Liste der gewünschten Sprachen
- accept-charset: Liste der beherrschten Zeichensätze
- referer: URL von der aus die Anfrage aufgerufen wird
- content-length: Länge des Inhalt-Teils in Bytes
- accept-encoding: Liste der Encodierungen, die der Anfragende beherrscht

**Antwort**

Die Antwort vom Server hat ein ähnliches Format:

```
<Protokoll> <Status-Code> <Status-Text>
<Header>

<Content>

```

Es gibt dabei eine Menge von Status Codes ([http.cat](https://http.cat/)), hier sind die gängigsten:

- 200 OK
- 301 Moved Permanently
- 302 Found 
- 400 Bad Request
- 401 Unauthorized
- 403 Forbidden 
- 404 Not Found
- 500 Internal Server Error
- 502 Bad Gateway
- 503 Service Unavailable

**Neuere HTTP Protokolle**

- HTTP/2
  - Kompression des Headers mit HPACK
  - Unterstützung für TLS ist verpflichtend
  - Push-Nachrichten
  - Optimierung des gleichzeitigen Ladens mehrerer Ressourcen
- HTTP/3
  - Wechsel von TCP zu QUIC
  - Vermeidung des Head-of-Line-Blocking
  - Generelle Verschlüsselung

HTTP/2 sieht man manchmal, HTTP/3 aber noch sehr selten.
