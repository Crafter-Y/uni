## Threads

**Parallelität**

Rechner ermittelt automatisch, ob einzelne Abarbeitungselemente gleichzeitig ausgeführt werden können und führt dies, falls möglich, für den Anwender und Programmierer transparen durch.

Um tatsächlich parallele Abläufe zu realisieren, erfordert es mehr als eine CPU, bzw, entsprechende Recheneinheiten (Kerne).

**Nebenläufigkeit**

Nebenläufige Ausführung ist auf eine einzige physische Maschine beschränkt, die jedoch mehrere vollständige CPUs enthalten kann.

Die Organisation erfolgt aber gesteuert durch den programmierer und i.d.R. nicht automatisch. In Java können mit `Threads` solche Unterprozesse erstellt werden.

**Prozess**

Ein Prozess wird durch das Betriebssystem als eigenständige Instanz – in der Regel unabhängig und geschützt von anderen – ausgeführt. Ein Prozess kann aber auch Unterprozesse erstellen und in diesen Daten ein- und ausgeben. Prozesse können auch untereinander kommunizieren.

**Threads**

Ein Thread stellt eine nebenläufige Ausführungseinheit genau eines Prozesses dar. Threads haben große Ähnlichkeiten mit Prozessen, wie Zustand, Programmzähler. Ein Thread in Java hat einen lokalen und einen geteillten Speicherbereich über welchen er sich mit dem Haupt-Thread austauschen kann. Dieses Verhalten bringt aber auch Risiken, wie z.B. Race-Conditions mit sich oder wenn eine Speichereinheit gleichzeitig gelesen und beschrieben wird. 

Warum Threads?

- Threads nutzen die vorhandenen Ressourcen moderner Betriebssysteme und Hardware effizienter aus
- Berechnungsaufwendige Arbeiten können "in den Hintergrund" verlagert werden.

Die Verwendung von Threads führt (auf einem Prozessor) im technischen Sinne nicht zu einer beschleunigten Ausführung. Durch das erstellen von Threads entsteht sogar noch ein Overhead, welcher tendenziell zu schlechterer Leistung führt. 

Erst beim Einsatz mehrerer Prozessoren in einer Maschine ergibt sich ein echter positiver Laufzeiteffekt durch die Möglichkeit, Threads auf verschiedenen CPUs zur Ausführung zu bringen.

**Threads in Java**

In Java stehen von Haus aus die folgenden Ansätze zur Verfügung: `Thread` und `Runnable`, wobei `Thread` intern auch auf `Runnable` aufbaut. 

Bei der Implementierung einer `Thread` Klasse, überschreibt man die `run` Methode von `Runnable` diese wird vom Laufzeitsystem aufgerufen, nicht vom Entwickler. Den erstellten Thread startet man mit der `start` Methode:

```java
public class FirstThreadDemo {
    public static void main(String args[]) {
        new Thread(() -> {
            for (int i = 0; i < 10; i++) {
                System.out.println(i + " ");
                try {
                    Thread.sleep(100);
                } catch (InterruptedException e) {
                    System.out.println(e);
                }
            }
            System.out.println("End of thread " + this.toString());
        }).start();

        System.out.println("End of main");
    }
}
```

Ausgabe:

```bash
> java FirstThreadDemo
End of main
0
1
2
3
4
```

Methoden von Thread:

- `static Thread currentThread()`: Referenz zum gerade ausgeführten Threadobjekt
- `ThreadGroup getThreadGroup()`: ThreadGroup, zu der dieser Thread gehört
- `static boolean holdsLock(Object)`: Prüft, ob der Thread ein Lock auf dem Objekt hat
- `interrupt()`: Unterbricht diesen Thread
- `boolean isAlive()`
- `join()`: Wartet auf das Beenden dieses Threads.
- `setDaemon(boolean)`: Markiert diesen Thread als Daemon oder User-Thread.
- `static void sleep(long)`: Lässt den Thread für die gegebenen Millisekunden schlafen
- `start()`: Startet die Ausführung dieses Threads.
- `static void yield()`: Lässt den gerade ausgeführten Thread (kurz) pausieren und gibt anderen Threads eine Chance zur Ausführung.
- `run()`: Wird von `start` aufgerufen. Wird die Methode nicht überschrieben, dann tut der Thread nichts.

**Synchronisation von Threads**

Die Methode `join` erlaubt es einem Thread, auf die Beendigung eines anderen Threads zu warten.

In vielen Fällen hat man aber nicht die Möglichkeit, jede Anweise zeilenweise zu steuern. 

In folgendender Datenklasse:

```java
class Name {
    private String firstname;
    private String surname;

    public void setName(String fn, String sn) {
        this.firstname = fn;
        this.surname = sn;
    }

    public String getName() {
        return this.firstname + " " + this.surname;
    }
}
```

Könnte ein Thread den Setter aufrufen, währenddessen ein anderer Thread die Getter Methode aufruft. In diesem Fall besteht die Möglichkeit, dass der `firstname` bereits überschrieben wurde und der `surname` noch nicht. Dadurch können kaputte Daten entstehen.

Mit dem `synchronized` keyword kann man das verhindern:

```java
class Name {
    private String firstname;
    private String surname;

    public synchronized void setName(String fn, String sn) {
        this.firstname = fn;
        this.surname = sn;
    }

    public synchronized String getName() {
        return this.firstname + " " + this.surname;
    }
}
```

`setName` und `getName` sind unteilbare Operationen geworden, d.h. solange eine Routine ausgeführt wird, kann keine andere `synchronized`-Routine ausgeführt werden.

**wait und notify**

Mit den global aufrufbaren Methoden `wait` und `notify` kann man Objektsperren auch manuell steuern:

- `wait()`: Thread geht in Zustand "wartend" über. Er gibt die Objekt-Sperre (Monitor) für andere Threads ab. 
- `notify()`: Erlaubt die wiederausführung eines anderen Threads, der ein `wait()` ausführt.
